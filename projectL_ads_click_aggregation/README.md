# ProjectL Ads Click Aggregation

Local implementation of an ads click event aggregation system with a Map/Aggregate/Reduce streaming architecture. It supports click redirects, durable raw click storage, derived minute-level metrics, idempotent click tracking, and log-line ingestion.

## What It Implements

- `GET /click` records a click and returns a `302` redirect to the advertiser URL.
- `POST /click/log` ingests click data from a single log line.
- `GET /metrics` returns advertiser click metrics at a minimum granularity of 1 minute.
- Raw click events are durably stored in SQLite as an append-only source of truth.
- Derived click metrics are stored separately in SQLite for low-latency queries.
- A streaming worker processes raw events through explicit Map, Aggregate, and Reduce nodes.
- Impression IDs are HMAC-signed and deduplicated so duplicate clicks are not counted.
- A reconciliation endpoint can rebuild derived metrics from raw events.

## Architecture

Click ingestion is intentionally separated from analytics processing:

1. The click endpoint validates the signed impression ID and target URL.
2. The service deduplicates by `impression_id` and appends accepted clicks to `raw_click_events`.
3. The Map node shards raw events by `ad_id`. Hot ads can be split into extra virtual shards.
4. The Aggregate node counts clicks by partition, ad, advertiser, and event-time minute.
5. The Reduce node combines partial counts into final advertiser/ad/minute deltas.
6. The stream processor upserts deltas into `derived_click_metrics` and advances its cursor in the same transaction.
7. Metrics queries read only the derived table, so advertiser queries stay fast.

This models Kafka/Kinesis-style stream retention with a local SQLite outbox. If the worker stops, raw events remain stored and are processed when it resumes.

## Duplicate Prevention

The system treats the ad placement's `impression_id` as the click idempotency key. An impression represents one rendered ad instance. If a client clicks the same rendered ad many times, every request carries the same `impression_id`, so only the first accepted click is counted.

The local implementation enforces this in `SQLiteClickStorage.record_click`:

1. The click service verifies the HMAC signature over `advertiser_id`, `ad_id`, and `impression_id`.
2. The raw click transaction first inserts the `impression_id` into `seen_impressions`.
3. `seen_impressions.impression_id` is a primary key. If the insert conflicts, the click is marked as a duplicate.
4. Only non-duplicate clicks are appended to `raw_click_events`.
5. Only raw events are processed by the Map/Aggregate/Reduce worker, so duplicates never reach derived metrics.

This prevents two common duplicate paths:

- A user double-clicks, refreshes, or retries the same click URL.
- A client or browser resubmits the same click request after a timeout.

### How Repeated Clicks Are Counted

The system deduplicates by impression, not by user and not by minute. A duplicate means "the same `impression_id` was clicked again." A new impression means the ad was rendered again and received a new `impression_id`.

Example 1: one user clicks the same rendered ad 5 times within one minute.

```text
minute=10:00 impression_id=imp_123 click #1 -> counted
minute=10:00 impression_id=imp_123 click #2 -> duplicate, ignored
minute=10:00 impression_id=imp_123 click #3 -> duplicate, ignored
minute=10:00 impression_id=imp_123 click #4 -> duplicate, ignored
minute=10:00 impression_id=imp_123 click #5 -> duplicate, ignored
```

The raw store receives only one accepted click event, so the derived metric for that ad and minute increases by `1`, not `5`.

Example 2: one user clicks once per day for three consecutive days.

If each click comes from a newly rendered ad, each render has a different `impression_id`, so all three clicks are counted:

```text
day=1 impression_id=imp_aaa click -> counted
day=2 impression_id=imp_bbb click -> counted
day=3 impression_id=imp_ccc click -> counted
```

Those clicks land in three different minute buckets based on event time. A daily metrics query would sum to `3` clicks across the three days.

If the user somehow reuses the exact same old click URL for all three days, the requests carry the same `impression_id`, so only the first click is counted and the later two are duplicates:

```text
day=1 impression_id=imp_aaa click -> counted
day=2 impression_id=imp_aaa click -> duplicate, ignored
day=3 impression_id=imp_aaa click -> duplicate, ignored
```

That behavior is intentional: the idempotency key protects against repeated submissions of the same ad impression. It does not try to enforce "one click per user per ad per day." That stricter rule would be a policy or fraud-control layer, which is outside this project's current scope.

The HMAC check prevents a client from fabricating unlimited new impression IDs. A valid click must include a signature generated by the ad placement side using the shared secret. The signature binds the `impression_id` to the specific advertiser and ad, so a valid impression for one ad cannot be replayed against another ad.

In production, `seen_impressions` would usually be a low-latency distributed dedupe store such as Redis Cluster, DynamoDB, or Cassandra with a TTL matching the attribution window. The important ordering is still the same: validate the signature, reserve the idempotency key, then publish or append the raw event. If the stream worker later replays raw events, the processor cursor and derived upserts keep aggregation idempotent at the batch boundary.

Reconciliation is the backstop. Since `raw_click_events` is the source of truth and contains only accepted clicks, the derived table can be rebuilt for a time range with `POST /reconcile` if a processing bug or partial worker failure produces incorrect aggregates.

## Hot Ad Sharding

The default Map node partitions events by `ad_id`, which keeps all clicks for an ad in the same logical shard. That is simple and efficient for normal traffic, but one very popular ad can create a hot shard. For example, if one ad gets thousands of clicks per second, a single partition, consumer, or aggregate task can fall behind while other shards are mostly idle.

The implementation supports virtual shards for hot ads through `MapNode.hot_ad_shards`:

```python
from app.stream import MapNode

map_node = MapNode(
    shard_count=32,
    hot_ad_shards={"ad_very_popular": 16},
)
```

For normal ads, the partition key is based on the stable hash of `ad_id`:

```text
shard_007:ad_123
```

For a configured hot ad, the Map node adds a virtual shard suffix selected from the `impression_id` hash:

```text
shard_007:ad_very_popular:0
shard_007:ad_very_popular:1
...
shard_007:ad_very_popular:15
```

This spreads one ad's click stream across multiple aggregate tasks. Each Aggregate node counts its local slice by advertiser, ad, and minute. The Reduce node then removes the physical partition concern by merging all partial aggregates back into one final key:

```text
(advertiser_id, ad_id, minute_start) -> click_count
```

That means advertisers still query one clean metric series for the ad. They do not need to know whether the ad was internally split across one shard or many virtual shards.

In a production Kafka or Kinesis deployment, the same pattern maps to the message partition key. Normal ads use `ad_id`; hot ads use `ad_id:virtual_shard`. The hot-ad configuration can be driven by recent click volume, campaign launch metadata, or spend tier. Once traffic cools down, future events can move back to fewer virtual shards because the Reduce node always emits the same advertiser/ad/minute metric key.

## Log Line Format

The log parser accepts JSON or key-value text.

```text
ts=2023-11-14T22:13:20Z advertiser_id=adv_1 ad_id=ad_1 impression_id=imp_1 target_url=https://advertiser.example/a signature=<hmac>
```

Required fields:

- `advertiser_id`
- `ad_id`
- `impression_id`
- `target_url`
- `signature`

Aliases such as `advertiser`, `adId`, `timestamp`, `url`, and `sig` are also accepted.

## Local Run

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
uvicorn app.api:app --host 127.0.0.1 --port 8080 --reload
```

Run the stream worker in another shell:

```bash
python -m app.worker --db data/ads_clicks.sqlite3
```

For local demos, you can also let the API schedule a small background processing pass after each click or call:

```bash
curl -X POST 'http://127.0.0.1:8080/stream/drain'
```

## Example Flow

Create an impression signature:

```bash
curl 'http://127.0.0.1:8080/sign?advertiser_id=adv_1&ad_id=ad_1&impression_id=imp_1'
```

Record a click and redirect:

```bash
curl -i 'http://127.0.0.1:8080/click?advertiser_id=adv_1&ad_id=ad_1&impression_id=imp_1&target_url=https%3A%2F%2Fadvertiser.example%2Fa&signature=<hmac>'
```

Query metrics:

```bash
curl 'http://127.0.0.1:8080/metrics?advertiser_id=adv_1&ad_id=ad_1&start_time=1699999980&end_time=1700000060&granularity_seconds=60'
```

Run tests:

```bash
pytest -q
python scripts/smoke.py
```

## Scale Notes

The local implementation uses SQLite so it can run without external infrastructure. In production, the same boundaries map cleanly to larger systems:

- Raw storage and stream retention: Kafka or Kinesis plus object storage archive.
- Map node: stream partitioning by `ad_id`, with virtual shards for hot ads.
- Aggregate node: per-partition minute counters.
- Reduce node: final merge into advertiser/ad/minute aggregates.
- Derived storage: ClickHouse, BigQuery, Snowflake, Redshift, or another OLAP store.
- Reconciliation: periodic batch rebuild from raw archived events.
